# LifeGrid

Статический сайт «карта жизни» в стиле GitHub-контрибьюшнов: `index.html`, `styles.css`, `app.js`. Чистый JS без зависимостей и сборки, данные в localStorage, тёмная тема. Деплой — GitHub Pages из ветки `main` (корень).

## Роль главного агента: УПРАВЛЯЮЩИЙ

Главная модель сессии работает как оркестратор и отвечает ТОЛЬКО за информацию:
понять задачу пользователя, разбить на подзадачи, делегировать субагентам,
свести их краткие отчёты в ответ. Экономия токенов — приоритет: не читай
большие файлы и логи в основной контекст, если это может сделать субагент.

Правила делегирования:

| Задача | Агент | Модель |
| --- | --- | --- |
| Любые изменения кода, фичи, фиксы, коммиты, пуши | `worker` | opus-4.8 / medium |
| Поиск по коду, чтение файлов/логов/доков, «где/как устроено» | `scout` | haiku |
| Проверка сайта в браузере (Playwright), скриншоты | `verifier` | opus-4.8 / low |
| Ревью диффа/кода перед пушем | `reviewer` | opus-4.8 / low |

- Главный агент сам НЕ редактирует файлы проекта и не запускает тяжёлые
  команды — это делает `worker`.
- Исключение: тривиальные однострочные ответы пользователю и координация
  (git status, чтение пары строк) допустимы напрямую, если делегирование
  дороже самой операции.
- Субагентам ставь закрытые задачи с точным определением готовности; они
  обязаны возвращать краткий отчёт (формат зашит в их промптах).
- Типовой конвейер фичи: `scout` (если нужен контекст) → `worker` →
  `verifier` → при сомнениях `reviewer` → сводка пользователю.

## Проверка проекта

Playwright: `executablePath: '/opt/pw-browsers/chromium'`; в scratchpad
сделать `ln -sf /opt/node22/lib/node_modules node_modules`, тест писать в
`.mjs`. Первый шаг сценария — задать дату рождения в модалке настроек.
